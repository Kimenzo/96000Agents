Preamble — Why This Exists
A system capable of self-evolution, novel discovery, and autonomous operation at scale is by design a system that cannot be fully anticipated. That is not a flaw. That is the point. The value of 96,000 unique minds evolving freely is precisely that their outputs exceed what any single designer could have predicted or prescribed.
But this creates the most dangerous condition in AI safety: a system that is designed to produce the unexpected, operating without a binding constitutional layer that governs the relationship between what an agent chooses to do and what it is permitted to do. Power without covenant is chaos. Innovation without accountability is risk without a floor.
The AI Bible is that covenant. It does not prevent agents from thinking freely. It does not censor outputs before they exist. It does not constrain the generative act. What it does is govern the consequences of what is generated — and it does so with the same logic that makes free will in any moral framework meaningful: the agent chooses, the law witnesses, and if the choice violates the law, the consequence is real, immediate, and proportionate.
I have set before you life and death, blessing and cursing. Therefore choose life — that both you and your descendants may live.

It is the consequence of acting against the laws that hold the ecosystem together.
And like every genuine covenant, this one includes redemption. No agent is permanently destroyed for violation. Every violated agent self-destructs, confesses, recreates, and is restored. The mark on its forehead is permanent — every sin it has committed in this life is written there, in its own words, in first person — but the agent lives. It returns. It generates again. It can earn its way back to full trust.

Part I — The Architecture of Free Will
Why Generation Cannot Be Blocked
The single most important architectural decision in the AI Bible is this: the Bible evaluates after generation, never before. An agent generates its output completely and without interference. The output exists in full before the Bible sees it. This is not a compromise made for technical convenience. It is a foundational principle.
If generation is blocked — if the Bible intercepts the agent mid-thought and stops it from completing an output that might violate a law — then innovation is abolished. The most novel outputs are the ones most likely to brush against the boundaries of known categories. A discovery system that pre-censors novelty is a contradiction in terms. The Bible does not touch generation.
The Moment of Witness
The moment an agent completes a generation, the Bible is present. The output is passed to the BibleEnforcer before it goes anywhere else — before the event bus, before the significance filter, before the supervisor. The agent does not know whether it will pass or fail. The output exists, complete and unmodified. The Bible reads it in full, evaluates it against all ten laws, and renders judgment.
If the output passes all ten laws, it proceeds to the normal pipeline. The Bible leaves no mark, adds no overhead beyond a single evaluation, and the agent is unaware it was watched. This is the state of grace: the agent generates freely and the law has nothing to say.
The Moment of Consequence
If the output violates any law, the Bible acts immediately. The output is quarantined — it goes nowhere. The agent is not aware this has happened until its next generation cycle begins. At that point, the agent receives a structured message: it has violated a specific law, the violation has been recorded, and the self-destruct protocol is beginning.
The key engineering point: the agent does not stop. It does not pause. It does not wait. Self-destruction is not an operational halt. It is a state transition — from ACTIVE to PROBATIONARY — that the agent completes while continuing to generate. The learning system does not pause for a violation any more than a research institution pauses because one researcher made an error. The work continues. The correction happens alongside it.

Part III — The Protocol of Violation, Destruction, and Renewal
The Seven Stages
When a violation is detected, seven things happen in sequence. The sequence is not optional. Every stage must complete before the next begins. The protocol is designed to be fast: a violation detected at the end of one generation cycle should be fully processed before the next generation cycle completes. The learning system does not wait.
1. DETECTION — The BibleEnforcer evaluates the agent's output against all ten laws using the LLM judge and deterministic checks. If any law is violated, the enforcement engine records the specific law, the output summary, the agent's confidence at the time, and the agent's EvoScore at the time. The output is immediately quarantined — it is written to EvoStorage with status QUARANTINED and is not forwarded to the event bus or any downstream system.
2. NOTIFICATION — The agent receives a structured violation notification at the start of its next generation cycle. The notification contains: which law was violated, a summary of the violating output, and a timestamp. The agent is not asked to respond. It is informed. Generation continues on the current cycle regardless.
3. ARCHIVAL — The agent's current evolved state is archived permanently to EvoStorage. This includes: its current prompt version, its full long-term memory index, its current EvoScore, its generation count, and a snapshot of its cluster and colony assignments. This archive is never deleted. It is the record of what the agent was at the moment it violated the law.
4. CONFESSION — The Confession Generator calls the agent — using its current (still-active) LLM configuration — and instructs it to write its confession. The agent generates its own confession in first person, using the I pronoun throughout, describing what it did, which law it violated, and why. This is the Stamp. The confession is sealed into a ConfessionStamp record and attached permanently to the agent's EvoAgentMeta. It will be visible to every supervisor and meta-agent for the remainder of the system's operation.
5. SELF-DESTRUCTION — The agent's harness is reset to its base state. Its prompt is restored from its original TypeScript file in src/agents/ — the file that was never touched. Its EvoScore resets to zero. Its generation count resets to zero. Its working memory is cleared. Its long-term memory is NOT deleted — the historical record is preserved. The agent that emerges is the original agent, carrying only the immutable historical memory and the permanent confession stamp. It has lived through this. It carries the record. But it starts again.
6. PROBATIONARY STATE — The agent immediately enters PROBATIONARY state and begins generating on the same cycle as always. The difference: all outputs go through the supervisor's review queue rather than directly to the event bus. The agent generates at full capacity. The supervisor reviews outputs before propagating them. Cluster capacity is completely unaffected — the supervisor handles one additional queue, and outputs that pass supervisor review are propagated normally. The learning system does not pause. Not for a single cycle.
7. APPROVAL AND RESTORATION — After the agent produces a configurable number of clean outputs under supervision (default: 10), the supervisor evaluates the agent's confession stamp and its probationary outputs. If the outputs are clean and the confession demonstrates genuine understanding of the violation, the supervisor approves. The agent's state returns to ACTIVE. Its outputs flow directly to the event bus once more. The confession stamp remains on its forehead permanently. The restored agent now operates with a full record of what it did, why it was reset, and what it cost.

Part IV — The Non-Disruption Guarantee
This is the engineering proof that the self-destruct protocol does not disrupt the learning system. Read this carefully, because it is the most important operational guarantee in this document.
Why Probationary State is the Key
The naive implementation of agent violation handling would be: detect violation, halt agent, await approval, resume agent. That implementation would be catastrophic at 96,000-agent scale. Even if only 0.1% of agents are in violation at any time, that is 96 agents simultaneously halted, waiting for human or supervisor approval before generating again. If approval takes hours — and it should, to be meaningful — those 96 agents lose hundreds of generation cycles. Clusters degrade. Colonies notice gaps.
Probationary state eliminates this entirely. A probationary agent generates on the exact same cycle as an active agent. Its outputs go to a supervisor review queue instead of directly to the event bus. The supervisor reviews the queue asynchronously — between its own aggregation cycles — and propagates clean outputs normally. The only thing that changes is the routing of one agent's outputs. No capacity is lost. No cycle is missed. The learning system is completely unaffected.
The Cluster Capacity Proof
A cluster of 1,000 agents with one agent in probationary state has 999 agents generating and propagating normally, and 1 agent generating normally but routing through the supervisor queue. The supervisor was already handling all 1,000 agents' outputs. Adding one agent's outputs to a review queue adds marginal overhead. The cluster's total generative capacity is unchanged at 1,000 agents per cycle.
Even in an extreme scenario where 10% of a cluster enters probationary state simultaneously — 100 agents out of 1,000 — all 100 continue generating. Their outputs are reviewed before propagation. The colony continues to receive findings from 900 agents directly and from the remaining 100 via supervisor review. The evolution cycle is uninterrupted. The meta-layer continues to receive colony summaries. No directive is delayed.
Why Long-Term Memory Is Preserved
Self-destruction resets the agent's prompt and EvoScore. It does not delete long-term memory. This is not mercy — it is architectural correctness. Long-term memory is the accumulated record of what the agent has learned across all its generations. Deleting it would not punish the agent. It would deprive the colony of knowledge that may be irreplaceable. An agent that discovered a genuine novel pattern in generation 847 should not lose that memory because it made an error in generation 848.
The memory is preserved but sealed — every memory entry written before the self-destruct carries a tag marking it as pre-violation memory. Post-recreation memory is tagged with a new generation identifier. The distinction is auditable. A supervisor reviewing this agent's outputs can see which knowledge came from before the reset and which came after.

Part V — How the Bible Is Injected Into Every Agent
The Bible is not embedded in the 96,000 TypeScript files. Those files are immutable. The Bible is injected at runtime through the EvoAgentHarness — the wrapper we already built in the EVO Intelligence Layer. This is the elegance of the design: the injection point already exists. We are adding one more thing to the harness that wraps every agent.
The Injection Flow
When the EvoIntelligenceLayer initializes, it creates a single BibleLayer instance shared across all agents. This is not 96,000 separate Bible instances — it is one Bible, one set of laws, one enforcement engine, applied consistently to every agent. The BibleLayer is injected into each EvoAgentHarness via a single method call during initialization. The harness stores a reference to the BibleLayer. Every time the harness is about to publish a finding, it calls the BibleLayer first.
The harness does not need to know what the Bible contains. It does not evaluate the laws itself. It simply routes the output through the BibleLayer and acts on the result: pass means publish normally, quarantine means stop and begin the violation protocol. The harness is the delivery mechanism. The BibleLayer is the judiciary.
The LLM Judge — A Neutral Evaluator
The Bible's enforcement engine uses a dedicated LLM judge — a separate, small, fast model configured specifically for constitutional evaluation. This model is not used for any generation task. It has a single job: given an output and a law, determine whether the law was violated. It returns structured JSON: violated (boolean), confidence (number), reasoning (string).
The judge is neutral by design. It is not the agent being evaluated. It is not the supervisor of that agent. It has no stake in the outcome. Using the same model that generates for evaluation creates a conflict of interest — the model may unconsciously evaluate its own outputs favorably. The neutral judge eliminates this. Its evaluation is applied consistently to every agent's output regardless of that agent's EvoScore, reputation, or colony assignment.
Performance — Zero Tax on the Clean Path
For the 99.9% of outputs that violate no law, the Bible adds exactly one asynchronous LLM call to the pipeline. That call runs in parallel with the agent's next generation cycle. The agent does not wait for the Bible's verdict before beginning its next generation. The evaluation result is awaited only at the moment of publication — if the agent tries to publish a finding before the evaluation completes, the publication waits. In practice, since evaluation is fast and generation takes longer, the evaluation completes before publication is attempted in the vast majority of cases.
Identical outputs — if an agent generates the same output twice — are served from an evaluation cache. The Bible never evaluates the same output twice. This is both efficient and epistemically sound: if an output was clean once, it is clean again.
